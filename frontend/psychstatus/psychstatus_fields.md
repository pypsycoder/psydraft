# PsyDraft (psychstatus) — техническая спецификация полей формы

Документ описывает все поля ввода формы психиатрического осмотра PsyDraft (`frontend/psychstatus/`) — идентификаторы, типы, варианты ответов и особенности поведения — для переноса формы в стороннюю медицинскую информационную систему (МИС). Раздел «Нозологический советник» (подсказка диагноза, `js/ui/nosology_advisor.js`) в документ не включён по требованию заказчика документа.

Разделы документа идут в порядке пунктов навигации (сайдбара) приложения.

---

## Общие технические примечания

1. **Движок мультивыбора (`js/ui/multicheck.js`).** Большинство групп чекбоксов реализованы одинаково: все `<input type="checkbox">` группы имеют одинаковый атрибут `name="<группа>"`, а `value` каждого чекбокса — это одновременно и машинное значение, и текст, который врач видит на экране (отдельного кода варианта нет). Часть групп обёрнута в раскрывающийся блок `multicheck-group` с заголовком и строкой-сводкой (`id="<группа>_summary"`, только для UI, не хранит данные). Функция `getCheckedTextByName(name)` в `multicheck.js` собирает все отмеченные значения группы через `", "` — именно так значения группы попадают в текст консультации.
2. **Вариант «другое».** В части групп есть чекбокс со служебным значением `value="D_OTHER"` (в тексте — «другое»). При его отметке появляется текстовое поле `id="<группа>_other"`; введённый в нём текст подставляется в итоговый список вместо слова «другое». Такой вариант есть только в группах `behavior`, `emotions` и (в собственной реализации) `appearance` — в остальных группах варианта «другое» нет.
3. **Пары «чекбокс + своё текстовое поле».** В нескольких блоках (иллюзии, типы галлюцинаций, идеи) каждый чекбокс — самостоятельный элемент с `data-*`-атрибутом, определяющим тип, и рядом с ним отдельное текстовое поле для описания именно этого варианта (а не общее поле группы).
4. **Хранение данных.** В текущей версии кода **не реализовано** персистентное сохранение состояния формы: `js/state/storage.js` содержит функции-заглушки `loadState()` (всегда возвращает `null`) и `saveState(state)` (ничего не пишет, просто возвращает переданный объект), `js/state/schema.js` (`FORM_SCHEMA = {}`) и `js/state/defaults.js` (`DEFAULT_STATE = {}`) пусты. То есть данные формы существуют только в DOM на момент нажатия кнопки «Сформировать текст»; localStorage реально не используется, вопреки общему описанию архитектуры проекта в CLAUDE.md. При интеграции с МИС значения нужно читать непосредственно из полей формы (по `id`/`name`), а не ожидать ключей localStorage.
5. **Итоговый текст.** Кнопка «Сформировать текст» вызывает `generateText()` (`js/report/builder.js`), которая собирает значения всех полей в готовый текст консультации и помещает его в `id="result"` (редактируемый `div`, не `<textarea>`). Логика самой сборки текста в документ не включена — она не нужна для маппинга исходных полей.

## 5. Рекомендации (`#section-recommendations`, `sections/recommendations.html` + `js/ui/recommendations.js`)

Раздел построен вокруг статичной библиотеки `RECOMMENDATIONS_LIBRARY` — фиксированного набора пунктов, каждый с `id`, группой (`group`) и текстом. Ввод врача сводится к отметке нужных пунктов чекбоксами; свободного текста в этой библиотеке нет (кроме итогового редактируемого поля).

### Механика UI

- `id="group-filters"` — кнопки-фильтры по группам (в т.ч. кнопка «Все»); влияют только на отображение, не на сохранённые данные.
- `id="recommendation-cards"` — контейнер, куда рендерятся элементы библиотеки.
- Для групп `general`, `sleep`, `anxiety`, `depression`, `psychosis`, `substance`, `extra` каждый пункт — **карточка**: чекбокс `class="rec-toggle" data-rec-id="<id>"` включает пункт; после включения становятся доступны радиокнопки `name="rec-<id>-mode"` со значениями `short` («Коротко») и `full` («Полно») — выбирают, какой вариант текста пункта попадёт в итог.
- Для групп `instrumental`, `laboratory`, `consultations`, `nonmedical` пункты выводятся **таблицей** (колонки: чекбокс, «Название», «Показание»); режима «Коротко/Полно» у них нет — используется только `title`.
- `id="recommendations-selected-count"` — счётчик отмеченных пунктов (только для UI).
- Кнопка `id="collect-recommendations-button"` («Собрать рекомендации в поле ниже») запускает сборку итогового текста.
- `id="recommendations"` — **редактируемый `div` (`contenteditable="true"`), не `<textarea>`** — итоговый список рекомендаций. Заполняется как нумерованный список: сначала обычные пункты (текст `short` или `full` в зависимости от выбранного режима карточки), затем — с заголовками разделов — списки отмеченных обследований/консультаций (только по названию, без «Показание»).
- Назначение препарата (раздел 6) может автоматически отмечать часть пунктов групп `instrumental`/`laboratory` (см. `RX_MONITORING_MAP` ниже) — это не действие врача, а автоматика формы.

## 6. Назначения (`#section-prescriptions`, `sections/prescriptions.html` + `js/ui/prescriptions.js`, `js/ui/rx_drug_dict.js`, `js/ui/rx_monitoring_dict.js`)

### 6.1 Форма добавления/редактирования назначения

Данные копятся не напрямую в DOM, а в JS-массиве в памяти `prescriptions` (элемент: `{ id, name, durationValue, durationType, times: { "утром"|"днём"|"вечером"|"на ночь": "доза_мг" }, comment }`); DOM-поля — только форма ввода одной записи за раз.

| id / name | Подпись | Тип | Варианты / особенности |
|---|---|---|---|
| `rx-name` | Название препарата | автокомплит-поиск (текст) | placeholder «например, Сертралин», `autocomplete="off"`. Поиск (`searchRxDrug()`) — подстрока (регистронезависимо) по МНН, торговому названию или любому аналогу из справочника `RX_DRUG_DICT` (см. 6.3), от 2 символов, до 25 результатов в `rx-drug-results`. Навигация стрелками/Enter, выбор подставляет в поле полную строку вида `МНН (Бренд, Аналог1, Аналог2…)` |
| `rx-duration-value` | В течение | select | Пусто (`—`) или целое число от 1 до 30 |
| `rx-duration-type` (name, радиогруппа) | (единица длительности) | радиокнопки | `days` — «дней» (выбрано по умолчанию), `months` — «месяцев» |
| 4 × `rx-time-checkbox` (`class`, `data-time`) | Схема приёма и доза (мг) | группа чекбоксов, каждый со своим полем дозы | `data-time`: «утром», «днём», «вечером», «на ночь» |
| 4 × `rx-dose-input` (`class`, `data-time`) | (доза, мг) | число | `min=0 step=1`, placeholder «мг»; поле активно только если отмечен соответствующий чекбокс времени (иначе `disabled` и очищается) |
| `rx-comment` | Комментарий | многострочный текст (textarea, 2 строки) | placeholder «например, принимать после еды» |
| `rx-clear-btn` | Очистить | кнопка | Сбрасывает форму и выходит из режима редактирования |
| `rx-submit-btn` | Добавить / Сохранить | кнопка | «Добавить» в режиме создания, «Сохранить» в режиме редактирования существующей записи; после отправки автоматически запускает проверку мониторинга препарата (см. 6.4) и пересборку блока рекомендаций |

Список добавленных назначений — `id="prescriptions_list"`, каждый пункт содержит текст (название, схема приёма, длительность, комментарий) и кнопки «✏️ Редактировать» / «❌ Удалить» (`data-id`, `data-action`).

### 6.4 Автоматический мониторинг терапии (`RX_MONITORING_MAP`, `js/ui/rx_monitoring_dict.js`)

Это не поле ввода, а бизнес-логика, срабатывающая после добавления назначения (`onRxSubmit()` → `applyDrugMonitoring(name)`): МНН препарата (часть строки `rx-name` до скобки, в нижнем регистре) ищется в карте `RX_MONITORING_MAP`. При совпадении:
- перечисленные в `items` идентификаторы пунктов из библиотеки рекомендаций (раздел 5, группы `instrumental`/`laboratory`) автоматически отмечаются как выбранные (режим `short`);
- если для препарата задан `alert`, его текст показывается врачу в блоке уведомления `id="rx-monitoring-notice"`.

Существенно для маппинга: это связь «препарат → предзаполненные пункты обследований», врач может её отменить (снять чекбокс) вручную. Ниже — полная карта.

|---|---|---|
| сертралин | exam_ecg | — |
| флуоксетин | exam_ecg | — |
